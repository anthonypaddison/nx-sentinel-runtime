## Commit 2 - Storage/Protocol Deduplication

- Added shared browser scoped-storage helpers in `config/www/nx-displaygrid/util/scoped-storage.util.js` for key naming, local JSON IO, prefix cleanup, and legacy migration.
- Refactored `prefs`, `persistence`, and `refresh` modules to use the shared storage helpers (no behavioral change intended).
- Added websocket command constants in `config/www/nx-displaygrid/util/ws-protocol.util.js` and used them in frontend persistence calls.
- Added Python websocket command constants in `config/custom_components/nx_displaygrid/websocket.py` and reduced duplicated setup code in `config/custom_components/nx_displaygrid/__init__.py`.
- Started tracking the `config/custom_components/nx_displaygrid/` backend integration files in git (they were hidden locally by a global ignore), excluding `__pycache__`.
- Added tests for scoped storage helpers and websocket command stability.
- Kept existing local frontend hardening edits intact while refactoring around them.

Why:
- Storage key/migration logic and websocket command names were duplicated across modules, increasing drift risk during schema changes and recovery workflows.
- The backend custom component must be versioned for reproducible deploys and rollback-safe setup migrations.

How to verify:
1. Run `node --test tests/*.test.mjs` and confirm all tests pass, including `scoped-storage` and `ws-protocol-commands`.
2. Run `python3 -m py_compile config/custom_components/nx_displaygrid/__init__.py config/custom_components/nx_displaygrid/websocket.py`.
3. In HA, load the dashboard and confirm persisted `nx-displaygrid` config still loads and saves (WS when backend is available, local fallback otherwise).
